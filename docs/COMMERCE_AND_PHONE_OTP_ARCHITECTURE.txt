1. BUSINESS MODEL
-----------------
The website serves two customer channels from one catalogue and one owner admin panel:

A. PERSONAL / B2C — Factory to Customer
- A person browses the public retail catalogue.
- Retail prices, retail pack sizes and customer availability are displayed.
- Phone OTP login is required before submitting an order request.
- A personal profile is created after first login.
- The customer can save addresses, review previous requests and track status.

B. BUSINESS / B2B — Factory to Wholesaler
- A business user authenticates through the same phone OTP system.
- A business profile and wholesale application are created after first login.
- The application contains business, GST and licence information required by the owner.
- The owner approves, rejects, requests correction, suspends or expires access.
- Wholesale prices remain hidden until approval.
- Approved wholesalers see carton quantities, MOQ, assigned price lists and wholesale-only products.

2. LOGIN ENTRY POINTS
---------------------
Public customer login:
- Route: /login
- Used by personal customers and business applicants/wholesalers.
- The page asks only for phone number and OTP.
- Account type is selected only for a first-time customer after successful OTP verification.

Direct admin login:
- Route: /admin/login
- Used only by pre-authorised owner/admin accounts.
- The admin enters an approved phone number and OTP.
- After verification, the server checks the owner_admin or staff role.
- A recognised owner_admin is sent directly to /admin.
- A recognised staff account is sent to the permitted admin area.
- A normal customer cannot become an admin by choosing an option on the screen.
- Unknown or non-admin accounts are denied access to admin routes.

The word Admin must not appear as a selectable public account type.
Admin roles are assigned only through protected database operations.

3. FIRST-LOGIN PROFILE ROUTING
------------------------------
After OTP verification, the server loads the authenticated user's profile and roles.

Routing order:
1. If role is owner_admin, redirect directly to /admin.
2. If role is staff, redirect to the authorised admin dashboard.
3. If an existing personal profile exists, redirect to /account or the previous retail destination.
4. If an existing business profile exists:
   - pending or correction_required: redirect to /business/application-status;
   - approved: redirect to /wholesale;
   - rejected, suspended or expired: redirect to the correct status/help page.
5. If no completed profile exists, show the account-type selection page.

First-time account-type choices:
- Personal — “I am buying for myself or my family.”
- Business — “I represent a shop, wholesaler or business.”

No Admin choice is shown.

4. PERSONAL PROFILE CREATION
----------------------------
A first-time personal customer provides:
- full name;
- verified phone number from Supabase Auth, read-only;
- optional email;
- city;
- state;
- preferred language;
- acceptance of privacy, safety and applicable service conditions.

System result:
- profiles.account_type = personal;
- user role = retail_customer;
- profile status = active;
- onboarding_completed = true.

A personal account can immediately use retail features, subject to location, stock and compliance checks.

5. BUSINESS PROFILE CREATION
----------------------------
A first-time business user provides:
- legal or trading name;
- contact person;
- verified phone number from Supabase Auth, read-only;
- optional email;
- business type;
- GSTIN where applicable;
- shop/office address;
- delivery address;
- state and intended service area;
- fireworks/explosives licence details where applicable;
- licence number and expiry date where applicable;
- supporting documents;
- expected purchase volume;
- acceptance of business, privacy and compliance declarations.

System result:
- profiles.account_type = business;
- user role = wholesale_applicant;
- business account status = pending_review;
- onboarding_completed = true after required fields are submitted.

A business applicant may use public retail access where permitted, but wholesale prices and wholesale ordering remain unavailable until owner approval.

6. ADMIN PROFILE AND DIRECT ACCESS
----------------------------------
Admin accounts are not self-registered.

Owner/admin setup:
- Create or verify the intended admin auth user securely.
- Assign owner_admin or staff in the protected user_roles table.
- Optionally keep an admin_allowlist of approved phone numbers for bootstrap protection.
- Record role assignment in audit_logs.
- Never allow a browser request to assign owner_admin to itself.

Admin login flow:
1. Open /admin/login.
2. Enter pre-authorised phone number.
3. Complete CAPTCHA/anti-bot check where configured.
4. Receive and verify OTP.
5. Server loads role from protected database data.
6. If owner_admin, create the authenticated session and redirect to /admin.
7. If staff, redirect according to permission scope.
8. Otherwise sign out or deny access and show a generic message.

Recommended production hardening:
- stronger rate limits for admin OTP;
- second factor or recovery method;
- recent-authentication requirement for high-risk actions;
- admin session timeout;
- audit every admin login and important change;
- optional IP/device alerts.

7. PHONE NUMBER AND OTP AUTHENTICATION
--------------------------------------
Authentication provider: Supabase Auth phone OTP.

Common login flow:
1. User enters an Indian or international phone number.
2. Application normalises it to E.164 format, for example +919876543210.
3. CAPTCHA/anti-bot validation runs before OTP request.
4. Supabase requests a six-digit OTP through the configured SMS provider.
5. User enters the OTP.
6. Supabase verifies it and creates a secure authenticated session.
7. Server-side profile and role routing runs.

Security requirements:
- OTP values are never stored in project tables, analytics or application logs.
- Rate-limit requests by phone number, IP and device signals.
- Add resend timer, expiry handling and maximum verification attempts.
- Use generic errors to reduce account enumeration.
- Block abusive phone numbers and IP addresses.
- Use secure cookies/session handling appropriate for the web application.
- Keep provider credentials only in Supabase or hosting secrets.
- Normal users cannot write their own role or approval status.

India SMS requirement:
- Complete applicable TRAI DLT Principal Entity, sender header and OTP content-template registration.
- Use a supported Supabase SMS provider or Send SMS Hook with a suitable Indian provider.

8. USER TYPES AND ACCESS
------------------------
- guest: published retail catalogue only.
- retail_customer: active personal profile with retail access.
- wholesale_applicant: business profile awaiting review or correction.
- wholesaler: owner-approved business account with assigned wholesale pricing.
- staff: optional restricted admin user.
- owner_admin: full catalogue, pricing, customer, wholesale, order and settings access.

Roles and profile ownership must be enforced through Supabase Row Level Security and trusted server checks, not only through hidden buttons.

9. COMPLIANCE-FIRST ORDERING RULE
---------------------------------
The application can support carts, request/order records, quotes, confirmations and status tracking.

Production behaviour remains configurable:
- ENQUIRY_ONLY: submit an enquiry/order request; no completed online sale.
- MANUAL_CONFIRMATION: owner verifies customer, licence where required, location, stock, transport and legal eligibility.
- LICENSED_ORDERING: enabled only after written legal/compliance confirmation for the exact model and service areas.

Default launch mode: ENQUIRY_ONLY / MANUAL_CONFIRMATION.

No payment gateway, automatic dispatch promise or completed online-sale wording is enabled merely because the software supports carts and orders.

10. PERSONAL / RETAIL CUSTOMER FLOW
-----------------------------------
1. Browse catalogue without login.
2. Add retail quantities to retail cart.
3. View estimated total and availability.
4. Login with phone OTP before submission.
5. Complete Personal profile on first login.
6. Add or select delivery/contact address.
7. Accept required safety, age, service-area and legal acknowledgements.
8. Submit order/enquiry request.
9. Store immutable product-name, price and quantity snapshots.
10. Owner checks stock, service area and compliance.
11. Customer receives confirmed amount and next steps.
12. Customer tracks status in My Orders.

11. BUSINESS / WHOLESALE CUSTOMER FLOW
--------------------------------------
1. Login with phone OTP.
2. Complete Business profile and wholesale application on first login.
3. Application status becomes pending_review.
4. Owner reviews information and private documents.
5. Owner may request correction, approve, reject, suspend or expire the account.
6. Approved wholesaler receives an assigned price list.
7. Only approved wholesalers see wholesale prices, MOQ, carton quantities and B2B availability.
8. Wholesaler submits quote/order request.
9. Owner confirms price, stock, documentation, transport and terms.
10. Approval, pricing and status changes are audited.

12. PRICING MODEL
-----------------
Each product may have:
- retail_price;
- retail_mrp;
- retail_min_quantity;
- wholesale price-list entries;
- wholesale_moq;
- carton_quantity;
- retail availability and visibility;
- wholesale availability and visibility.

Flexible model:
- price_lists: Retail, Wholesale Standard, Wholesale Silver, Wholesale Gold or custom.
- product_prices: product, price list, unit price, minimum quantity and effective dates.
- business_accounts.price_list_id: assigned wholesale price list.

Displayed and submitted prices must be recalculated on trusted server/database logic. Browser-submitted prices are never trusted.

13. CART AND ORDER RULES
------------------------
- Separate retail and wholesale carts.
- Retail and wholesale items cannot be mixed in one request/order.
- Out-of-stock, hidden and ineligible products cannot be submitted.
- Quantity must follow pack, MOQ and carton rules.
- Orders store immutable product and price snapshots.
- Inventory reservation begins only after owner confirmation unless explicitly configured otherwise.

Suggested statuses:
- draft;
- submitted;
- compliance_review;
- awaiting_stock_confirmation;
- quote_sent;
- customer_accepted;
- confirmed;
- processing;
- packed;
- dispatched;
- delivered;
- rejected;
- cancelled.

16. ROW LEVEL SECURITY SUMMARY
------------------------------
- Guests read only published retail catalogue fields.
- Personal customers read/change only their own profile, addresses, cart and orders.
- Business applicants read only their own business profile, application and status.
- Approved wholesalers read only assigned wholesale pricing and their own business/order data.
- A wholesaler never reads another wholesaler's pricing, documents or orders.
- Public users cannot read phone numbers, wholesale documents or admin data.
- Only protected owner/admin operations change products, prices, roles, approvals and order status.
- Users cannot promote account_type into an admin role.
- Service-role credentials never appear in browser code.

18. ACCEPTANCE CRITERIA
-----------------------
- A first-time customer authenticates with phone OTP.
- A first-time customer chooses Personal or Business after OTP verification.
- Personal selection creates an active retail profile.
- Business selection creates a business profile and pending wholesale application.
- Admin is not available as a public selection.
- A pre-authorised owner_admin logs in at /admin/login and is routed directly to /admin.
- A normal customer cannot access admin routes even with a valid OTP.
- Returning users bypass account-type selection and reach the correct destination.
- Wholesale pricing remains hidden before owner approval.
- Every role and record is protected by tested RLS/server checks.
- No application code may trust a client-supplied role, approval status or price.
- Production mode cannot silently turn an enquiry into an online sale.
